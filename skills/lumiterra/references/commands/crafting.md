# Crafting commands — crafting

> Command interface source of truth: the "Crafting commands:" section in `src/parser/help.js` + `src/parser/validators/crafting.js`
> Command list: `craft-execute` / `query-recipes` / `query-item-sources`

Crafting commands cover recipe query and execution: `query-recipes` lists craftable recipes, `query-item-sources` reverse-looks up ways to obtain an item, and `craft-execute` crafts an item by recipeId at a crafting NPC. All commands return JSON; failures return `{ "success": false, "errors": [...] }`.

> **Drift from help.js**: `query-craft-lottery` and `craft-lottery` are already registered in `bin/lumiterra.js` (the `validators` mapping table), but have not yet been written into the "Crafting commands:" section in `help.js`. Code (`bin/lumiterra.js` + `validators/craft-lottery.js`) is the source of truth.

## `craft-execute`

**Purpose**: craft items by recipe. Internally executes automatically: navigate to crafting NPC -> interact -> submit craft -> wait for completion -> extract finished item. This is a long-running command and can be interrupted by `stop`.

**Parameters**:

| flag | Required | Default | Description |
|---|---|---|---|
| `--recipe` | Yes | - | Recipe ID (from `query-recipes.recipes[].id`) |
| `--count` | Yes | - | Craft count (1-10) |
| `--buildingId` | No | - | Specify crafting building ID (if omitted, automatically finds nearest usable building) |

**Example**:

```bash
lumiterra craft-execute --recipe 12001 --count 3
```

**Return fields**:

- Success: crafting result (including produced item list)
- Failure: `data.stage` indicates which step got stuck
- `stage` enum: `navigate` (failed to navigate to NPC) / `interact` (interaction failed) / `craft_submit` (craft submission failed, commonly insufficient materials / not unlocked / queue full) / `craft_wait` (wait timeout) / `craft_extract` (failed to extract finished item)

**Failure example**:

```json
{
  "success": false,
  "data": {"stage": "craft_submit"},
  "errors": ["Craft failed: insufficient materials (need Iron Ore x5, current x2)"]
}
```

**Notes**:

- Preconditions: enough materials, recipe unlocked, crafting queue not full
- Before crafting, first use `query-recipes --craftable` to filter recipes that can be made, avoiding insufficient materials at `craft_submit`
- If materials are insufficient, use `query-item-sources --item-cid <materialCid>` to query acquisition routes
- This is long-running; `stop` can interrupt within 2s

## `query-craft-lottery`

**Purpose**: query all energy-potion lottery tiers and current energy-potion balance. Read this before lottery and use it with `craft-lottery`.

**Parameters**: none.

**Example**:

```bash
lumiterra query-craft-lottery
```

**Return fields**:

- `energyBalance` (number): current energy-potion count
- `lotteries[]`: tier list
  - `recipeId` (number): tier ID, pass to `craft-lottery --recipe-id`
  - `name` (string): tier name (for example `"Lv 3 energy potion lottery"`)
  - `timeCost` (number): wait seconds per draw
  - `cost` (array): materials consumed per draw `[{itemCid, num}]`
  - `rewards` (array): reward pool `[{itemCid, num, quality}]`

**Notes**:

- `lotteries[].recipeId` is a lottery tier ID, **not** a recipe ID. Do not mix it with recipeId from `craft-execute --recipe` / `query-recipes`.
- Read `energyBalance` before drawing to decide whether there is enough balance.

## `craft-lottery`

**Purpose**: execute one batch of energy-potion lottery draws (single or multiple). The Unity side opens the lottery UI and triggers the UI single-draw / ten-draw flow.

**Parameters** (validator: `validateCraftLottery`):

| flag | Required | Default | Description |
|---|---|---|---|
| `--recipe-id` | Yes | - | Tier ID (positive integer, from `query-craft-lottery.lotteries[].recipeId`) |
| `--count` | No | 1 | Draw count for this batch, range `1-10` |

CLI-side validation (`validators/craft-lottery.js`):

- Missing or non-positive-integer `--recipe-id`: reports `"Missing required parameter: --recipe-id (positive integer)"`
- Non-integer `--count` or outside range `1-10`: reports `"--count must be between 1 and 10"`
- After validation, both `recipe-id` and `count` are converted to strings before being sent

**Examples**:

```bash
lumiterra craft-lottery --recipe-id 90001
lumiterra craft-lottery --recipe-id 90001 --count 10
```

**Return fields**:

- `recipeId` (number): tier ID
- `recipeName` (string): tier name
- `count` (number): draw count for this run
- `batchId` (string | null): single-draw batch ID; `null` for multi-draw
- `batchIds` (array): list of batch IDs generated by this draw
- `rewards` (array): obtained rewards `[{itemCid, num, quality}]`

**Failure scenarios**:

- `data.stage` enum: `pre_check` (pre-check failed, such as insufficient energy) / `open_ui` (failed to open lottery UI) / `ui_draw` (UI draw flow failed)
- Extra fields when energy is insufficient: `maxDrawable` (max draw count with current energy), `energyBalance` (current balance)

**Notes**:

- `--recipe-id` is the lottery tier ID, not the item recipe ID.
- When energy is insufficient, first read returned `maxDrawable` to decide whether to downgrade or replenish energy; do not blindly retry.
- This is a UI-driven flow. On failure, use `stage` to locate whether it is a local UI issue or server rejection.

## `query-recipes`

**Purpose**: query recipe list. By default, only unlocked recipes are returned, and `lockedRecipeCount` tells how many locked recipes under the current query conditions were hidden. Locked recipes can be included when needed.

**Parameters**:

| flag | Required | Description |
|---|---|---|
| `--recipe-id` | No | Specify recipe ID; returns it even if locked |
| `--include-locked` | No | Also return locked recipes (default false) |
| `--level` | No | Return only the specified level |
| `--talent-type` | No | Specialization: `battle`/`farming`/`gather` (aliases `combat`/`farm`/`gathering`) |
| `--craftable` | No | Return only recipes with enough materials (default false) |

**Example**: `lumiterra query-recipes --talent-type battle --level 12 --craftable`

**Return fields**:

- `recipes[]`: `{id, name, level, talentType(battle|farming|gather|unknown), unlocked, unlockSource(default|talent_tree), talentTreeId, timeCost, materials[]{itemCid,need,have}, products[], maxCraftable}`
- `includeLocked` (automatically true when `--recipe-id` is passed), `totalCount`, `lockedRecipeCount`
- `totalCount`: number of `recipes[]` returned by this query.
- `lockedRecipeCount`: after applying query conditions such as `--level` / `--talent-type` / `--craftable`, the number of matching but locked recipes; when `includeLocked=false`, this means the number of hidden locked recipes.

**Note**: prefer `--recipe-id` / `--level` / `--talent-type` / `--craftable` to narrow precisely, and avoid global filtering unless necessary. If the default result cannot find the target and `lockedRecipeCount > 0`, first use `--include-locked` or `--recipe-id` to confirm whether the recipe is merely locked; do not directly conclude "no recipe".

## `query-item-sources`

**Purpose**: query all acquisition routes for an item. Agents use this command to connect the decision chain "recipe -> materials -> sources".

**Parameters**:

| flag | Required | Description |
|---|---|---|
| `--item-cid` | Yes | Item config ID |

**Example**: `lumiterra query-item-sources --item-cid 1234`

**Return fields**:

- `itemCid`, `itemName`
- `sources[]`: `{type, sourceId, sourceName}`

**`sources[].type` enum**:

| type | Meaning | `sourceId` meaning |
|---|---|---|
| `monster_drop` | Kill monsters for drops | monsterCid -> `auto-combat --target` |
| `fishing` | Fishing | fishCid |
| `craft` | Crafting | recipeId -> `craft-execute --recipe` |
| `seed_planting` | Planting and harvest | seedCid |
| `gathering` | Gathering (tree chopping / mining / grass cutting) | resourceCid |
| `animal_pet` | Animal husbandry: pet animals to harvest | animalCid -> first `animal-query --cid` to get `entityId`, then `animal-pet` |
| `pickup` | Random field pickup, **no usable spawn point** | Not applicable (see notes below) |

**Notes**:

- Warning: when querying spawn points, `query-spawn-point --cid` must pass the `sourceId` in the table above (monsterCid/resourceCid/seedCid/animalCid/fishCid), **not** this command's input `--item-cid` (finished itemCid).
- `craft` type has no spawn point; go directly through the recipe flow.
- `pickup` type: **MUST NOT** call `query-spawn-point`; when `pickup` appears in `sources[]`, the agent **MUST** report a blocker (no stable acquisition route).
- When there is no source, `sources` is an empty array, not an error.
